---

### 🌱 專案起源背景

我想要把醫學影像操作變簡單——這就是 pydicomRT 最初的念頭。要把 DICOM 處理成可以實際操作的物件，往往得先深入研究儀器的成像原理與資料型態，其中 RT（放射治療）的需求特別複雜。

我在使用現有的開源工具處理 RTSTRUCT 時發現幾個痛點：
- **輪廓資料難以與 3D 影像對齊** — 座標系統之間的轉換缺乏一致性實作
- **Mask ↔ Contour 轉換不直覺** — 臨床軟體用 Mask，工程研究用 Contour，中間轉換往往是客製化 script
- **IOD 驗證不足** — 自行拼湊的 RTSTRUCT 容易違反 DICOM 標準，上游系統一讀就爆炸

所以我決定從頭打造一個專門處理 RTSTRUCT 的 Python 套件，讓研究人員用幾行 code 就能操作這些複雜資料，不必自己踩坑。

---

### 📋 變動內容

這是 pydicomRT 的第一個功能 commit，我建立了核心套件，包含：

- **RTSTRUCT 建立與解析** — 可以從頭創建新的 RTSTRUCT 檔案
- **3D Mask ↔ Contour 轉換** — 兩邊格式的相互轉換
- **座標系統轉換** — Image space ↔ Patient space
- **IOD 驗證** — 確保符合 DICOM 標準
- **範例腳本** — 附有使用範例

18 個檔案、1642 行新增代碼，是一個相當完整的起始點。

### 💭 熊熊的開發心得

**Q1: RTSTRUCT 處理第一版最困難的部分是什麼？**
> 說來可能意外——最難的不是 coding，而是規劃。因為我不確定日後會冒出什麼新需求，在整體架構上特別小心，生怕設計得太死。最後我決定根據 DICOM 資料的類型去設計各式各樣的處理方法，讓子模組各自發展時不會互相綁死。

**Q2: Contour Sequence 的格式在網路上資料多嗎？還是主要靠 DICOM 標準文件？**
> 網路上的資料很少，大部分還是得靠 DICOM 標準文件自己啃。另一部分則是我參考了 rt-utils 的實作方式，幫助理解官方文件中那些難以消化的描述。

**Q3: 當時有參考其他 Python DICOM 處理庫嗎？pydicomRT 和它們的定位有何不同？**
> 最一開始我確實用 rt-utils 處理 RTSTRUCT 相關資料，但後來發現一些更細節的寫入方式不在這個套件的範圍內——像是 Contour data 的浮點數精度、更詳細的 ROI 設定等。pydicomRT 的定位就是補上這些 rt-utils 照顧不到的地方，變成一個更完整的 RT 專門工具。
